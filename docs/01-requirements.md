# 微信智能日程规划 Agent 系统需求文档

## 1. 项目概述

本项目是一个基于独立微信通道的轻量多用户智能日程规划 Agent 系统。系统以微信为主要交互入口，用户通过微信自然语言输入日程、任务、提醒、学习计划、会议安排等内容，系统由 Agent 进行理解、拆解、规划、冲突检测和提醒，并通过微信持续交互。

系统同时提供 Web Dashboard，作为日程驾驶舱，用于展示每日计划、日历视图、任务池、冲突事项、提醒任务、Agent 执行日志、微信绑定、邀请码和系统配置。

本项目不是传统 Web 日历系统，也不是简单提醒机器人，而是：

```text
微信通道驱动
+
SchedulePlanningGraph
+
日程 / 任务 / 计划工具集
+
确定性冲突检测
+
微信提醒
+
Web 日程驾驶舱
+
邀请码轻量多用户体系
```

一句话定位：

```text
基于独立微信通道的轻量多用户智能日程规划 Agent 系统。
```

重要边界：本项目不使用外部 Agent Runtime，不使用外部 Agent 编排。微信连接、授权、消息收发由自研微信通道服务承担，真正的 Agent、业务、权限、数据库和 Web API 全部由自研 FastAPI 后端实现。

## 2. 项目目标

系统需要实现以下目标：

1. 用户可以通过微信自然语言创建固定时间日程。
2. 用户可以通过微信自然语言创建弹性任务。
3. 用户可以查询今天、明天、本周或指定日期的安排。
4. Agent 可以根据用户任务和已有日程生成每日计划。
5. 系统可以检测时间冲突、任务过载、截止日期风险等问题。
6. Agent 可以针对冲突事项给出调整建议。
7. 复杂规划必须先生成草案，用户确认后再写入正式计划。
8. 系统可以通过微信进行到点提醒。
9. Web 端可以展示完整日程、每日计划、任务池、冲突事项和 Agent 执行过程。
10. owner 可以生成邀请码，member 通过邀请码注册并使用系统。
11. member 拥有完整个人日程 Agent 功能，但不能访问系统设置、全局日志、模型配置、用户管理等管理功能。

## 3. 用户角色

系统支持轻量多用户，但不是复杂企业级多租户系统。

系统至少包含两类用户：

```text
owner
member
```

### 3.1 owner 主用户

owner 是系统主用户和系统拥有者，拥有最高权限。

owner 可以：

1. 使用微信 Agent 管理自己的日程。
2. 创建、查询、修改、删除自己的日程。
3. 创建弹性任务。
4. 生成每日计划。
5. 查看和处理自己的冲突事项。
6. 接收微信提醒。
7. 使用完整 Web 日程驾驶舱。
8. 生成邀请码。
9. 查看邀请码列表和使用情况。
10. 禁用邀请码。
11. 管理被邀请用户。
12. 禁用或恢复 member 用户。
13. 查看和修改系统设置。
14. 配置 LLM 模型。
15. 查看微信通道状态。
16. 查看全局消息日志。
17. 查看全局 Agent 执行日志。
18. 查看全局提醒任务。

### 3.2 member 被邀请用户

member 是通过邀请码加入系统的普通用户。

member 可以：

1. 使用微信 Agent 管理自己的日程。
2. 创建固定时间日程。
3. 创建弹性任务。
4. 查询自己的日程。
5. 生成自己的每日计划。
6. 检测自己的冲突事项。
7. 接收自己的微信提醒。
8. 使用 Web 日程驾驶舱查看自己的计划。
9. 查看自己的任务池。
10. 查看自己的冲突事项。
11. 查看自己的提醒任务。
12. 修改自己的个人偏好设置。

member 不能：

1. 查看或修改系统设置。
2. 查看 LLM API Key。
3. 修改模型配置。
4. 查看微信通道全局连接配置。
5. 查看其他用户的日程、任务、提醒、计划和冲突。
6. 查看全局消息日志。
7. 查看全局 Agent 执行日志。
8. 查看用户管理页面。
9. 生成系统级邀请码，除非后续 owner 授权。
10. 禁用或管理其他用户。
11. 查看敏感运行日志。

## 4. 微信接入方式

系统使用自研微信通道服务作为微信消息接入通道。

微信通道职责：

```text
接收微信文本消息
发送微信文本回复
发送微信提醒
维护微信通道连接状态
```

系统不自行实现 Wechaty、GeWeChat 或个人号 Bot Gateway。微信连接、授权、消息收发由自研微信通道服务承担。

首版支持：

```text
私聊文本消息
文本回复
文本提醒
```

首版暂不支持：

```text
群聊
图片
语音
文件
位置
富媒体卡片
复杂微信菜单
```

如果未来支持群聊，则只处理 @Agent 的消息，避免误触发。

## 7. 核心业务流程

### 7.1 用户注册与邀请码

owner 可以在 Web 中生成邀请码。

邀请码包含：

```text
邀请码
创建者 user_id
可使用次数
已使用次数
过期时间
状态
备注
创建时间
更新时间
```

邀请码状态：

```text
active
used_up
expired
disabled
```

member 注册流程：

```text
用户访问 Web 注册页
  ↓
输入邀请码
  ↓
系统校验邀请码
  ↓
创建 member 用户
  ↓
记录邀请码使用记录
  ↓
用户进入自己的 Web 日程驾驶舱
```

### 7.2 微信身份绑定

每个系统用户需要通过二维码登录绑定自己的微信身份。通道账号凭证由 `wechat_accounts` 保存，用户与微信身份映射由 `channel_identities` 保存。

绑定流程：

```text
用户登录 Web
  ↓
创建微信二维码登录会话
  ↓
用户使用微信扫码并确认
  ↓
系统轮询登录状态并获取通道凭证
  ↓
写入 `wechat_accounts` 和 `channel_identities`
```

未绑定用户发送普通消息时，系统回复：

```text
你还没有绑定账号，请先在 Web 中创建二维码登录会话并完成微信确认。
```

### 7.3 创建固定日程

用户输入：

```text
明天下午 3 点开项目会议，提醒我
```

系统流程：

1. 自研微信通道服务接收微信消息。
2. WeChat Channel Adapter 标准化消息。
3. 系统根据 `conversation_id` 或 `channel_user_id` 找到 `user_id`。
4. SchedulePlanningGraph 判断用户意图为 `create_event`。
5. Agent 提取标题、日期、时间、提醒需求。
6. Agent 通过 Tool Executor 调用 `create_event`。
7. 系统创建 `calendar_event`。
8. 系统创建 `reminder_job`。
9. 系统执行冲突检测。
10. Agent 生成自然语言回复。
11. 通过微信返回创建结果。

回复示例：

```text
已为你创建日程：项目会议，时间为明天下午 3 点。我会在开始时提醒你。
```

### 7.4 创建弹性任务

用户输入：

```text
明天我要写论文 2 小时，帮我安排一下
```

系统流程：

1. Agent 判断这是一个弹性任务。
2. 创建 `task_item`，记录预计耗时 120 分钟。
3. 查询明天已有日程。
4. 查找可用空闲时间。
5. 生成计划草案。
6. 保存 `agent_pending_state`。
7. 通过微信请求用户确认。

回复示例：

```text
我查看了你明天的安排，建议把“写论文”安排在 09:00-11:00。

明天计划草案：
09:00-11:00 写论文

是否确认写入计划？回复“确认”即可。
```

### 7.5 生成每日计划

用户输入：

```text
帮我安排明天的计划
```

系统流程：

1. 查询明天已有固定日程。
2. 查询未完成且适合明天安排的任务。
3. 检测可用时间段。
4. 根据优先级、截止日期、预计耗时安排任务。
5. 检测计划是否过载。
6. 生成 `day_plan` 和 `plan_item` 草案。
7. 通过微信返回计划草案。
8. 用户确认后写入正式计划。

### 7.6 冲突检测

用户输入：

```text
明天下午 3 点开会，帮我加一下
```

如果已有：

```text
明天 15:00-16:00 项目讨论
```

系统应检测到冲突并回复：

```text
发现时间冲突：

你明天 15:00-16:00 已有“项目讨论”，与新日程“开会”时间重叠。

建议：
1. 将新会议改到 16:30-17:30
2. 保留新会议，同时标记冲突
3. 取消创建新会议

请回复 1/2/3。
```

### 7.7 查询日程

用户输入：

```text
明天有什么安排？
```

回复示例：

```text
你明天有以下安排：

09:00-11:00 写论文
15:00-16:00 项目会议
19:30-20:30 健身

目前没有发现冲突。
```

### 7.8 修改日程

用户输入：

```text
把明天下午 3 点的会议改到 4 点
```

系统流程：

1. Agent 判断意图为 `update_event`。
2. 调用 `search_event_candidates` 查找候选日程。
3. 如果唯一匹配，则调用 `update_event`。
4. 更新相关 `reminder_job`。
5. 重新检测当天冲突。
6. 返回修改结果。

### 7.9 删除日程

用户输入：

```text
取消明天下午 3 点的会议
```

系统流程：

1. Agent 判断意图为 `delete_event`。
2. 查找候选日程。
3. 用户确认或唯一匹配后删除。
4. 取消对应 `reminder_job`。
5. 重新计算当天计划。

### 7.10 到点提醒

当 `reminder_job` 到达触发时间：

1. Reminder Worker 扫描到期任务。
2. 生成提醒消息。
3. 通过 WeChat Channel Adapter 调用微信通道服务发送微信消息。
4. 更新 `reminder_job` 状态。
5. 如果是周期任务，生成下一次 `reminder_job`。

提醒示例：

```text
提醒：15:00 项目会议即将开始。
```

## 8. Agent 能力要求

SchedulePlanningGraph 是系统核心，不应只是关键词匹配器。

Agent 需要具备以下能力：

```text
自然语言理解
中文时间表达解析
固定日程识别
弹性任务识别
多事项拆解
每日计划生成
冲突检测调用
工具调用决策
用户确认管理
连续对话管理
自然语言回复生成
```

Agent 不应直接操作数据库，必须通过 Tool Executor 调用工具完成业务操作。

Agent 工具调用时必须自动注入当前 `user_id`，不允许跨用户查询或修改数据。

## 10. 冲突检测规则

冲突检测不应完全依赖大模型。LLM 负责理解和建议，程序负责确定性检测。

冲突类型包括：

```text
time_overlap
too_many_tasks
deadline_risk
insufficient_free_time
missing_time
ambiguous_intent
```

## 11. 用户确认机制

以下场景可以自动写入：

```text
单条明确固定日程
单条明确提醒
明确删除唯一匹配事项
明确修改唯一匹配事项
```

以下场景必须确认：

```text
一次输入多个事项
自动生成每日计划
存在时间冲突
存在多个候选日程
需要调整已有日程
需要批量写入计划
```

## 12. 时间处理要求

系统必须支持中文时间表达：

```text
今天
明天
后天
今晚
明早
上午
下午
晚上
下周一
周五下午
每天晚上
每周一上午
3 小时后
30 分钟后
本周
下周
这个月
```

默认时区：

```text
Asia/Shanghai
```

规则：

1. 如果用户没有指定年份，默认使用当前年份。
2. 如果用户说“明天”，必须基于系统当前日期计算。
3. 如果时间已经过去，应提示用户重新指定未来时间。
4. 如果任务没有固定时间但有预计耗时，应作为弹性任务进入任务池或计划草案。
5. 如果用户没有说明任务耗时，Agent 应追问或使用用户默认设置。

## 16. 开发阶段规划

### 阶段一：Agent 能力闭环

通过 Debug API 验证 Agent 能识别日程并完成创建、查询、修改、删除、任务规划、冲突检测和 pending state。

### 阶段二：Web Dashboard

展示 Agent 创建的数据和执行过程。

### 阶段三：微信通道接入

将微信通道接收到的微信消息接入同一个 SchedulePlanningGraph。

## 17. MVP 验收标准

1. owner 可初始化并登录。
2. owner 可生成邀请码。
3. member 可通过邀请码注册。
4. member 数据与其他用户隔离。
5. Debug API 输入“明天下午 3 点开会”可创建日程。
6. Debug API 输入“明天有什么安排？”可查询日程。
7. Debug API 输入“明天写论文 2 小时，帮我安排一下”可生成计划草案。
8. 用户回复“确认”可确认计划。
9. 存在时间重叠时可检测冲突并提示选项。
10. Reminder Worker 可到点触发提醒。
11. Web 可展示日程、任务、计划、冲突、提醒和 Agent 日志。
12. 微信绑定后，微信消息可进入同一个 Agent 流程。
13. 到点后微信可收到提醒。

## 18. 关键约束

1. 微信是主入口，不是附属通知渠道。
2. 微信通道只作为消息收发通道。
3. 不使用外部 Agent Runtime。
4. 不使用外部 Agent 编排能力。
5. Web 是日程驾驶舱，不是主业务入口。
6. Agent 是主脑，不是关键词判断器。
7. Agent 不直接写数据库，必须通过工具调用。
8. LLM 不负责精确冲突判断，程序负责确定性检测。
9. 复杂规划必须先生成草案，用户确认后写入。
10. 所有用户数据必须按 user_id 隔离。
11. Agent 工具调用必须自动注入当前 user_id。
12. Reminder 不经过 LLM。
13. 开发顺序必须 Agent 先行，消息通道后置。

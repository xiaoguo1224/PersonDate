# Web Dashboard 全面响应式适配设计

## 背景

当前 Web Dashboard 已具备完整业务页面，但整体布局仍以桌面端为中心，主要问题集中在以下几个方面：

1. 侧边栏在小屏设备上占用过多空间。
2. 顶部信息区在手机端容易拥挤，天气、问候语和操作按钮会挤压主内容。
3. 今日安排、日历、甘特图、日志列表等页面在窄屏下容易出现内容压缩或横向溢出。
4. 部分页面依赖大屏宽度才能舒展开，平板和窄桌面尺寸下体验不稳定。

本设计的目标不是重做一套移动端产品，而是在保留现有桌面驾驶舱风格的前提下，让同一套页面完整适配手机端、平板和不同尺寸的电脑端。

## 设计目标

1. 手机端可完整使用核心功能，至少覆盖 `320px` 到 `414px` 常见宽度。
2. 平板端保持良好可读性与可操作性，避免卡片和表单过密。
3. 桌面端保留当前“驾驶舱”感，不牺牲信息密度与多栏布局能力。
4. 角色权限逻辑不变，owner/member 的菜单隔离继续由后端权限和前端菜单共同保障。
5. 避免为移动端单独拆出一套页面，尽量用统一布局和响应式规则解决。
6. 仅对必要的页面结构和全局壳层做改造，不引入大范围视觉重构。

## 非目标

1. 不更换前端技术栈。
2. 不替换 Ant Design。
3. 不修改后端 API 设计。
4. 不新增原生 App 或小程序形态。
5. 不对业务文案和权限模型做额外调整。

## 方案选择

本次采用“桌面侧栏 + 手机顶部 Tab 栏 + 更多菜单”的混合方案。

原因如下：

1. 桌面端可以延续当前侧边栏导航，符合驾驶舱定位。
2. 手机端更适合高频入口集中在顶部，减少横向占位。
3. 通过“更多菜单”收纳次级功能，可以兼顾可发现性与简洁度。
4. 该方案对现有页面改动相对可控，风险低于彻底重构导航体系。

## 总体布局

### 桌面端

桌面端保留左侧侧边栏、顶部信息条和内容区三段式布局：

1. 左侧侧边栏展示主导航和分组菜单。
2. 顶部信息条展示日期、天气、问候语、快捷操作和用户头像。
3. 内容区使用自适应宽度，允许各页面按自身布局栅格排布。

### 手机端

手机端改为顶部固定 Tab 栏 + 更多菜单：

1. 顶部 Tab 栏只保留最核心的高频入口。
2. “更多”用于承载次级页面、owner 管理入口和账户相关入口。
3. 内容区上移，尽量保证首屏信息密度。
4. 顶部天气和问候语默认收敛为简短信息，必要时隐藏次要内容。

### 平板端

平板端介于桌面和手机之间，采用“压缩桌面壳层”的策略：

1. 侧边栏保留，但宽度收紧。
2. 顶部按钮和信息区允许换行。
3. 内容区优先保证可读性，不强行维持多栏。

## 响应式断点

建议采用以下断点策略：

| 断点 | 布局策略 |
|---|---|
| `>= 1280px` | 完整桌面布局，侧边栏常驻，信息密度最高 |
| `768px ~ 1279px` | 侧边栏收紧，内容区优先，部分页面由双栏降为单栏 |
| `< 768px` | 切换为手机顶部 Tab 栏 + 更多菜单 |
| `<= 390px` | 进一步压缩顶部信息与按钮，保证不溢出 |

## 导航设计

### 桌面侧边栏

桌面侧边栏继续承载完整功能分组，沿用现有 owner/member 权限分离。

### 手机顶部 Tab 栏

手机端只展示 4 个入口：

1. 今日安排
2. 日历视图
3. 任务
4. 更多

其中：

1. 前 3 个入口覆盖大多数日常操作。
2. “更多”打开一个收纳完整功能的菜单面板。
3. 当前路由应高亮对应 Tab。
4. 切换页面后，菜单自动收起。

### 更多菜单

更多菜单负责收纳低频功能和角色相关入口：

1. `owner` 继续显示用户管理、邀请码管理、系统设置、模型配置、微信通道状态、全局消息日志等。
2. `member` 仅显示自己的 Agent 记录、微信绑定、我的账号设置等。
3. 更多菜单建议使用单独面板承载，避免手机端下拉层过窄影响点击。

## 页面级适配策略

### 今日安排页

今日安排页是首屏核心页面，优先做最细致的响应式适配。

适配原则：

1. 桌面端保留主内容 + 右侧辅助栏的双栏布局。
2. 平板端可降级为上下堆叠，但保留内容分组。
3. 手机端改为单列卡片流，避免左右并排造成信息挤压。
4. 总览指标在手机端优先显示核心数字，次要说明折叠为小字。
5. 时间线、任务、冲突和提醒列表统一改为可阅读的纵向卡片。

### 日历视图页

日历页是最容易受宽度限制影响的页面，必须做视图级降级：

1. 月视图保留 7 列结构，但在窄屏下缩小单元格内容，事件数量以摘要形式展示。
2. 周视图和甘特图保留横向滚动能力，不强制压缩到不可读。
3. 手机端默认优先展示更适合阅读的视图，允许用户再切换到周视图或甘特图。
4. 视图切换区使用紧凑的分段按钮或标签栏，防止按钮换行后破坏布局。

### 任务页

任务页通常包含筛选、列表和操作按钮，移动端需要特别注意：

1. 桌面端可保持列表或表格形态。
2. 手机端优先采用卡片列表，减少列数。
3. 筛选条件收进折叠区域或抽屉面板。
4. 批量操作和次级操作收进“更多操作”菜单。

### 冲突事项页

冲突事项页以信息可读性为主：

1. 冲突条目在窄屏下改为单列摘要卡。
2. 冲突严重程度、涉及事项、建议动作分层展示。
3. 处理按钮在手机端纵向排列，避免横向拥挤。

### 提醒任务页

提醒任务页以状态列表和操作为主：

1. 手机端保留状态标签、时间和错误原因。
2. 长文本失败原因可折叠展示。
3. 取消、测试触发等按钮优先放入条目底部。

### Agent 日志页

日志页信息密度高，手机端必须降噪：

1. 默认只展示时间、摘要、状态和核心结果。
2. 工具调用和工具结果折叠展开。
3. 长 JSON 或长文本仅在展开后查看。

### 微信绑定、账号设置和系统管理页

这类页面以表单和设置项为主：

1. 表单在手机端一律单列显示。
2. 输入框、选择器、开关和按钮占满可用宽度。
3. 复杂确认操作使用全屏或接近全屏的弹层。
4. 风险操作按钮与普通按钮保持明显区分。

## 内容区与栅格规则

### 宽度控制

内容区不应无限拉伸到超宽屏幕边缘，建议采用统一的最大宽度控制：

1. 普通页面维持流式宽度。
2. 复杂页面允许更宽，但需要保留左右内边距。
3. 超宽屏下内容居中，避免信息过散。

### 卡片与分栏

卡片和分栏需要遵循以下规则：

1. 优先使用 `grid` 和 `minmax()` 做自适应。
2. 当可用宽度不足时，双栏自动降为单栏。
3. 多列卡片在中屏降为 2 列，在手机降为 1 列。
4. 不依赖固定像素宽度决定布局核心逻辑。

### 文本规则

1. 标题允许换行，但不应破坏操作区。
2. 次要说明优先截断或折叠。
3. 长标签和长时间字符串要提供换行或省略策略。
4. 所有重要文本必须保持可读，不靠 hover 才能理解。

## 交互与触控规则

1. 所有可点击元素在手机端应保证足够大的触控热区。
2. hover 仅作为增强，不作为信息唯一来源。
3. 菜单、弹窗和下拉层应避免在窄屏下贴边裁切。
4. 高风险操作尽量在确认层中二次确认。
5. 页面切换后保持滚动位置和当前视图状态的稳定性。

## 主题与视觉适配

当前主题系统保留，但响应式下需要额外处理两点：

1. 手机端减少重型背景动画和过强的装饰层，以免影响可读性与性能。
2. 在 `prefers-reduced-motion` 场景下减少过渡与粒子动画强度。
3. 主题切换按钮在小屏上应收纳进更紧凑的区域，不单独占用过多宽度。

## 无障碍与可用性

1. 保证键盘导航可达。
2. 保证当前选中态清晰可见。
3. 保证颜色对比度不因主题和缩放而失真。
4. 保证手机端不出现因固定定位导致的遮挡问题。

## 实施顺序

建议按以下顺序推进：

1. 先改造 `DashboardShell`，把桌面壳层和手机壳层分开。
2. 再补全手机顶部 Tab 栏与更多菜单。
3. 然后统一内容区响应式栅格和间距。
4. 接着逐页修正今日安排、日历、任务、冲突、日志和管理页。
5. 最后做交互细节、性能和可访问性修补。

## 验收标准

以下条件全部满足时，视为响应式适配完成：

1. 手机端 `320px`、`375px`、`390px`、`414px` 宽度下可正常使用。
2. 平板端 `768px`、`1024px` 宽度下无明显拥挤或截断。
3. 桌面端 `1366px`、`1440px`、`1920px` 下布局稳定。
4. 非预期横向滚动仅出现在允许滚动的日历周视图、甘特图和超宽表格区域。
5. owner/member 菜单隔离不被破坏。
6. 重要页面在手机端不需要横向缩放即可操作。
7. 弹层、抽屉和表单在小屏下不遮挡关键控件。

## 测试建议

1. 先做浏览器尺寸检查，覆盖手机、平板、常见笔记本和大屏桌面。
2. 重点验证导航切换、菜单展开和页面高频操作。
3. 逐页验证今日安排、日历、任务、冲突、提醒和日志页。
4. 验证 owner/member 菜单与路由权限一致。
5. 验证长文本、空状态、加载态和错误态在小屏下不溢出。
6. 验证主题切换后响应式规则仍然有效。

## 结论

本次响应式适配采用统一壳层、桌面侧栏保留、手机顶部 Tab 栏 + 更多菜单的方案。

它能在不破坏当前业务结构和权限体系的前提下，覆盖手机、平板和不同尺寸桌面，并且为后续页面细化提供一致的布局基线。
